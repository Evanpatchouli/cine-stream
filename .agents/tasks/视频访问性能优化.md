# 本地视频站公网访问性能优化评估文档

## 背景

当前项目是一个本地视频浏览站点：

- 视频资源存储在本机电脑硬盘中；
- 后端必须运行在本机，用于读取本地视频文件；
- 前端通过 frp 挂载到公网，朋友可以访问页面；
- 后端接口也需要通过 frp 暴露给公网；
- 当前本机在新疆，云服务器在上海，公网访问视频时链路较长，播放体验可能较慢。

当前链路大致为：

```text
朋友浏览器
  ↓
上海云服务器 frps
  ↓
新疆本机 frpc
  ↓
本机后端读取视频文件
  ↓
返回视频流
```

由于视频源站在新疆本机，frp 只能转发流量，不能消除跨省链路和本机上传带宽限制。因此需要从服务端视频传输方式、视频格式、缓存策略三个方向优化。

2. 视频转为 HLS 分片播放；
3. 云服务器 Nginx 缓存视频资源或 HLS 分片。

# 方案一：后端支持 HTTP Range 请求

## 目标

让浏览器播放视频时可以按需请求视频片段，而不是一次性下载整个视频文件。

浏览器播放视频时通常会发送类似请求头：

```http
Range: bytes=0-1048575
```

后端应该返回：

```http
HTTP/1.1 206 Partial Content
Accept-Ranges: bytes
Content-Range: bytes 0-1048575/123456789
Content-Type: video/mp4
Content-Length: 1048576
```

## 需要 Codex 检查的内容

请检查当前后端的视频接口是否存在以下问题：

- 是否一次性读取整个视频文件，例如 `readAllBytes()`；
- 是否直接返回完整文件流，未处理 `Range` 请求头；
- 是否返回了错误的 HTTP 状态码，例如始终返回 `200 OK`；
- 是否缺少：
  - `Accept-Ranges: bytes`
  - `Content-Range`
  - 正确的 `Content-Length`
  - 正确的 `Content-Type`

- 拖动视频进度条时，是否会重新请求整个文件；
- 大视频播放时是否会占用过高内存；
- 是否支持常见视频格式：
  - mp4
  - webm
  - mkv，如果前端播放器支持
  - m3u8，如果后续支持 HLS

## 推荐实现方向

如果是 Java / Spring Boot 后端，优先评估：

- Spring MVC `ResourceRegion`
- `InputStreamResource`
- `FileSystemResource`
- 自定义 Range 解析逻辑
- 静态资源映射是否能天然支持 Range

不要使用一次性读取整个文件的方式。

## 验收标准

使用浏览器或 curl 测试：

```bash
curl -I http://localhost:8793/api/videos/xxx
```

应看到：

```http
Accept-Ranges: bytes
```

测试 Range 请求：

```bash
curl -v -H "Range: bytes=0-1023" http://localhost:8793/api/videos/xxx
```

应返回：

```http
HTTP/1.1 206 Partial Content
Content-Range: bytes 0-1023/文件总大小
Content-Length: 1024
```

前端验收：

- 视频可以快速开始播放；
- 拖动进度条后不会重新下载整个视频；
- 大视频不会导致后端内存明显升高；
- 多人同时播放时后端不会因为整文件读取而卡死。

# 方案二：视频转为 HLS 分片播放

## 目标

将大视频文件转为 HLS 格式，让播放器按小分片加载视频，提高弱网和长链路场景下的播放体验。

原始视频：

```text
movie.mp4
```

转码后：

```text
movie_hls/
  index.m3u8
  segment_000.ts
  segment_001.ts
  segment_002.ts
  segment_003.ts
```

或者使用 fMP4：

```text
movie_hls/
  index.m3u8
  init.mp4
  segment_000.m4s
  segment_001.m4s
```

## 优点

- 每次只加载几秒的视频片段；
- 首屏播放更快；
- 拖动进度条更友好；
- 分片失败后重试成本更低；
- 后续可以接入 Nginx 缓存；
- 可以生成多码率版本，例如：
  - 1080p
  - 720p
  - 480p

## 缺点

- 需要额外转码或切片；
- 占用额外磁盘空间；
- 需要维护原视频与 HLS 文件之间的映射关系；
- 如果视频很多，不适合一次性全部转码；
- 需要前端播放器支持 HLS。

## 需要 Codex 检查的内容

请结合当前项目评估：

- 当前视频是否是直接播放 mp4；
- 前端播放器是否支持 HLS；
- 如果是原生 `<video>`：
  - Safari 可以原生播放 HLS；
  - Chrome / Edge 通常需要 hls.js；

- 是否需要引入 `hls.js`；
- 当前视频数据模型是否能记录：
  - 原始文件路径
  - HLS 输出目录
  - m3u8 地址
  - 转码状态
  - 分辨率
  - 时长
  - 文件大小
  - 封面图

- 是否需要后台转码任务；
- 是否需要只对热门视频或手动选择的视频进行 HLS 转换；
- 是否需要保留原始 mp4 直链播放作为兜底方案。

## ffmpeg 示例

单个视频转 HLS：

```bash
ffmpeg -i input.mp4 \
  -c:v libx264 -preset veryfast -crf 23 \
  -c:a aac -b:a 128k \
  -hls_time 6 \
  -hls_playlist_type vod \
  -hls_segment_filename "output/segment_%03d.ts" \
  output/index.m3u8
```

生成 720p HLS：

```bash
ffmpeg -i input.mp4 \
  -vf "scale=-2:720" \
  -c:v libx264 -preset veryfast -crf 26 \
  -c:a aac -b:a 96k \
  -hls_time 6 \
  -hls_playlist_type vod \
  -hls_segment_filename "output_720p/segment_%03d.ts" \
  output_720p/index.m3u8
```

## 推荐实现策略

不要一开始就全量转码所有视频。

推荐按阶段实现：

### 第一阶段

- 保留当前 mp4 播放；
- 后端先支持 Range；
- 选几个常看的视频手动转成 HLS；
- 前端支持播放 m3u8。

### 第二阶段

- 增加“生成 HLS”按钮；
- 后端调用 ffmpeg 转换单个视频；
- 记录转码状态；
- 转换完成后前端优先播放 HLS。

### 第三阶段

- 支持批量转码；
- 支持低码率版本；
- 支持任务队列；
- 支持失败重试；
- 支持删除 HLS 缓存文件。

## 验收标准

- m3u8 文件可以正常访问；
- ts 或 m4s 分片可以正常访问；
- Chrome / Edge 可以通过 hls.js 播放；
- Safari 可以直接播放或兼容播放；
- 拖动进度条可以快速加载对应分片；
- 弱网下比直接 mp4 更容易开始播放；
- 后续可以被 Nginx 缓存。

# 方案三：云服务器 Nginx 缓存视频资源或 HLS 分片

## 目标

在上海云服务器上增加 Nginx 反向代理和缓存，让重复访问的视频资源尽量从云服务器缓存返回，减少反复回源到新疆本机。

目标链路：

```text
朋友浏览器
  ↓
上海云服务器 Nginx
  ↓
命中缓存则直接返回
  ↓
未命中缓存才通过 frp 回源到新疆本机后端
```

## 适合缓存的内容

优先缓存：

- HLS `.ts` 分片；
- HLS `.m4s` 分片；
- 视频封面图；
- 字幕文件；
- 静态图片；
- 不频繁变化的视频元信息。

谨慎缓存：

- `.m3u8` 文件；
- 需要鉴权的接口；
- 用户相关接口；
- 播放记录接口；
- 搜索接口；
- 后台管理接口。

如果是 VOD 点播场景，`.m3u8` 也可以短时间缓存，例如 10 秒到 1 分钟。
如果视频文件不会变化，也可以更长。

## Nginx 缓存示例

示例配置，仅用于评估，实际路径和端口需要结合项目调整：

```nginx
proxy_cache_path /var/cache/nginx/video_cache
    levels=1:2
    keys_zone=video_cache:200m
    max_size=50g
    inactive=7d
    use_temp_path=off;

server {
    listen 8080;

    location /api/ {
        proxy_pass http://127.0.0.1:8793/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    location /videos/ {
        proxy_pass http://127.0.0.1:8793/videos/;

        proxy_cache video_cache;
        proxy_cache_valid 200 206 7d;
        proxy_cache_lock on;

        proxy_set_header Host $host;
        proxy_set_header Range $http_range;
        proxy_set_header If-Range $http_if_range;

        add_header X-Cache-Status $upstream_cache_status;
    }

    location ~* \.(ts|m4s|mp4|webm|jpg|jpeg|png|webp|vtt)$ {
        proxy_pass http://127.0.0.1:8793;

        proxy_cache video_cache;
        proxy_cache_valid 200 206 7d;
        proxy_cache_lock on;

        add_header X-Cache-Status $upstream_cache_status;
    }

    location ~* \.m3u8$ {
        proxy_pass http://127.0.0.1:8793;

        proxy_cache video_cache;
        proxy_cache_valid 200 1m;
        proxy_cache_lock on;

        add_header X-Cache-Status $upstream_cache_status;
    }
}
```

## 需要 Codex 检查的内容

请结合项目评估：

- 当前公网入口是否直接暴露 frp remotePort；
- 是否适合改为：
  - 公网用户访问 Nginx；
  - Nginx 反代到本机 frp 暴露的后端端口；

- 后端视频 URL 是否可以统一走 `/videos/` 或 `/media/` 前缀；
- HLS 分片路径是否稳定；
- 是否存在鉴权逻辑；
- 如果有鉴权，缓存是否会导致越权访问；
- 是否需要给视频资源增加签名 URL；
- 是否需要限制缓存目录大小；
- 是否需要定期清理缓存；
- 是否需要通过响应头观察缓存命中状态。

## 推荐缓存策略

如果项目目前没有复杂鉴权：

```text
/api/             不缓存
/videos/          可以缓存
/hls/             强烈建议缓存
/covers/          可以缓存
/subtitles/       可以缓存
```

如果项目有登录鉴权：

```text
/api/             不缓存
/videos/          谨慎缓存
/hls/             需要确认 URL 是否带权限隔离
/covers/          可以缓存
/subtitles/       视情况缓存
```

HLS 推荐：

```text
.m3u8             短缓存，10s ~ 60s
.ts / .m4s        长缓存，7d 或更久
封面图            长缓存
字幕              长缓存
```

## 验收标准

第一次访问视频分片时：

```http
X-Cache-Status: MISS
```

第二次访问同一个分片时：

```http
X-Cache-Status: HIT
```

如果是 HLS：

- 同一个 `.ts` 或 `.m4s` 分片被第二次访问时应命中缓存；
- 不应该每次都回源到本机；
- 多个朋友看同一个视频时，本机上传压力应明显下降。

3. 支持手动生成 HLS；

## P3：公网优化

1. 上海服务器增加 Nginx 反代；
2. 对 HLS 分片做缓存；
3. 对封面、字幕等静态资源做缓存；
4. 使用 `X-Cache-Status` 验证缓存命中。

# 约束

请不要直接大规模重构。

请先做评估，并给出最小可行改造方案。

优先保证：

- 本机后端仍然可以直接访问本地硬盘视频；
- 局域网访问不受影响；
- frp 公网访问可用；
- 前端播放体验改善；
- 不要引入过重的系统；
- 不要一次性要求全量转码所有视频；
- 不要破坏现有视频目录结构。

推荐路线：

```text
先 Range
再 HLS
最后 Nginx 缓存
```
